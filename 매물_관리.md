# 매물 관리

매물 데이터를 다루거나 새 매물을 탐색·추천할 때 참고한다.
데이터 단일 소스는 `data/listings.json`(매물)과 `data/config.json`(열 순서·분류)이다. `scripts/listings.py`는 이 JSON을 읽어 렌더링·검증 로직만 제공한다.

## 매물 항목 원칙 (필수)

**각 매물 항목의 이름과 정보는 실재하는 구체적인 부동산 매물(단지 하나)을 가리켜야 한다.** 항목명과 모든 정보(세대수·연식·전용면적·시세 등)가 **단일 단지로 특정**되어야 한다.

- `OO동 구축`·`OO마을 (A·B·C)`처럼 여러 단지를 묶은 **클러스터·일반 서술은 금지**한다. 세대수 같은 단지 고유값이 `단지별 500~715`처럼 범위로 적혀 있으면 특정 단지가 아니라는 신호다.
- 특정 단지로 확정할 수 없으면 **항목을 추가하지 않는다.** 이미 있는 부정확한 placeholder는 특정 단지로 분해하거나 **삭제**한다.
- **시세는 추정치여도 되지만**(실매물 대조 필요), **단지명·세대수·연식·전용면적은 추정이 아니라 실제 단지 값**이어야 한다.

## 분류 기준 (거래유형 → 시 → 구)

표는 3단계로 나뉜다: **거래유형**(`config.json`의 `groups`) 아래를 **'시' 단위**(`groups[].cities`)로, 다시 그 아래를 **'구' 단위**(`groups[].cities[].districts`)로 나눈다. 표는 구마다 1개씩 생긴다(예: 매매 > 서울시 > 관악구 > 표).

### 거래유형 (1단계)

- **매매**: 전세·월세가 아닌 매수 매물 전부(과거 '실거주'와 '재건축 투자'를 통합한 것). 거주 목적이든 투자 목적이든, 연식·평수와 무관하게 매수면 여기에 둔다.
- **월세/전세**: 주거에 드는 비용을 최대한 줄이고, 남는 여유 자금을 다른 상품에 투자하기 위한 매물(전세·월세).
  - 추천 조건(사용자 확정): 면적 **20평 이상**, **아파트(400가구+) 또는 오피스텔**, **보증금 1~3억**, **월 주거비(월세+관리비) 150~200만원**. 같은 매물이면 전세·월세 가격을 **모두 병기**.
  - **통근 기준(2026-06-20 사용자 확정, 종전 '외곽 허용' 기준 폐기)**: 목적은 '주거비 최소화 + 여유자금 투자'지만, 통근은 **정자역 대중교통 1시간 이내를 반드시 준수**한다(정자역이 구속 조건, 판교역은 가능하면 고려하는 보조 기준). 지역 범위도 매매와 동일하게 **서울·성남·용인·수원**으로 한정한다.
  - 월세/전세 매물도 반드시 **특정 단지 하나**로 적는다(위 [매물 항목 원칙] 준수). 보증금·월세·전세가 같은 **시세는 추정치여도 되나** 네이버 부동산 등 **실매물 대조가 필요**하며(데이터·정리본에도 명시), 단지명·세대수·연식은 실제 단지 값으로 채운다.

#### 매매 '목적' 컬럼 (실거주 / 몸테크)

매매 표에만 있는 `목적` 컬럼은 **연식**으로 기계적으로 정한다(사용자 확정 기준):

- **연식 2002년 이후(2002년 포함)** → `실거주`
- **연식 2001년 이전** → `몸테크`

연식은 `listings.json`의 `연식` 필드(`YY.MM` 표기, 예: `95.07`=1995년, `02.02`=2002년) 앞 두 자리로 판단한다(`YY≥50`이면 19xx, 아니면 20xx). Notion 표에서 `목적` 셀은 값에 따라 색이 입혀진다(실거주=파랑, 몸테크=빨강 — `make_notion.py`의 `PURPOSE_COLORS`). 새 매매 매물을 추가하거나 연식을 고치면 이 기준으로 `목적`을 다시 맞춘다.

### 시 (2단계)

각 거래유형 아래 매물을 **'시' 단위**로 묶는다(예: `경기도 성남시`, `경기도 용인시`, `경기도 수원시`, `서울시`).

- 기준 단위는 **'시'**다. 시 이름은 시·도를 함께 적는다(`경기도 용인시`). 서울은 `서울시`.
- **분당·판교(분당구)·야탑·오리·구미동은 성남시**, 수지·기흥·상현·흥덕·보정은 용인시, 권선구·영통은 수원시처럼 구·동 이름이 시를 직접 안 가리키는 경우 행정구역상 '시'로 환산해 배치한다.
- **매물이 있는 시만** 노출된다(빈 시는 만들지 않음). 새로운 시의 매물이 추가되면 그 시가 새로 생긴다.

### 구 (3단계)

각 시 아래를 다시 **'구' 단위**로 표를 나눈다. **표는 구마다 1개**다(예: `서울시 > 관악구`, `서울시 > 중랑구`).

- 매물이 속한 구는 매물의 `지역` 필드(예: `서울 관악구 봉천동`, `수원 영통 망포동`)로 판별해 `config.json`의 해당 `districts[].district`에 배치한다. 구 이름은 `관악구`·`분당구`처럼 **'구'까지** 적는다.
- 분당구·수정구(성남), 기흥구·수지구(용인), 권선구·영통구(수원), 관악구·중랑구(서울)처럼 한 시 안에 여러 구가 있으면 구별로 표가 따로 생긴다.
- **매물이 있는 구만** 표가 생긴다(빈 표는 만들지 않음).

#### 시·구 표 노출 순서

- **시 순서**: 거래유형 안에서 시가 나오는 순서는 **`config.json`의 배열 순서와 무관하게** `listings.py`의 `CITY_PRIORITY`가 결정한다(매매·월세/전세 양쪽 공통). 우선순위에 든 시를 지정 순서대로 먼저 놓고, 나머지는 가나다순으로 노출한다.

  1. 서울시
  2. 경기도 성남시
  3. 경기도 용인시
  4. 경기도 수원시
  5. 이하 나머지 시는 가나다순

  > 우선순위를 바꾸려면 `scripts/listings.py`의 `CITY_PRIORITY` 리스트를 고친다. 완성형 한글은 유니코드 코드포인트 순서가 곧 가나다순이라, 우선순위 밖 시들은 `sorted()`로 자동 정렬된다. config 배열을 손으로 재정렬할 필요 없이 push할 때마다 규칙이 적용된다.

- **구 순서**: 한 시 안에서 구 표가 나오는 순서는 별도 우선순위 없이 **`config.json`의 `districts` 배열 순서**를 그대로 따른다. 구 순서를 바꾸려면 해당 시의 `districts` 배열을 재배열한다.

## 매물 탐색·추천 시 고려사항

Claude가 새 매물을 탐색하거나 추천할 때 반드시 반영할 가구 조건/제약.

- **가구 구성**: 2인 무자녀 가정.
- **직장 위치 / 통근**: 두 사람의 직장이 각각 아래와 같다.
  - 분당선 **정자역** 근처
  - 신분당선 **판교역** 근처
  - 두 직장 모두 대중교통 기준 **출퇴근 1시간 이내**를 선호한다.
  - 단, 매물이 **서울시**에 위치한 경우 **1시간 30분까지 허용**한다.
- **예산**: **11억원 이내**.
- **'아파트' 기준**: 최소 **세대수 400가구 이상**인 단지만 '아파트'로 간주한다.

> 추천 시 위 조건을 모두 점검할 것: 정자역·판교역 양쪽 통근 시간, 예산 11억 이내, 세대수 400가구 이상. 서울 매물만 통근 상한이 1시간 30분으로 완화된다.

### 토지거래허가구역(토허제) 현황 — 2025.10.15 대책

2025년 10월 15일 주택시장 안정화 대책으로 토허제가 대폭 확대됐다(효력 2025.10.20~**2026.12.31**).

- **지정 범위**: **서울 25개 자치구 전역** + 경기 12곳(과천, 광명, **성남 분당·수정·중원구**, **수원 영통·장안·팔달구**, 안양 동안구, **용인 수지구**, 의왕, 하남).
- **규제**: 매수 시 구청 **허가 필수**(무허가 계약 무효), **취득 후 2년 실거주 의무**(이 기간 매매·임대 금지), 아파트뿐 아니라 연립·다세대도 포함. 무주택·처분조건부 1주택 **LTV 40%**(15억 이하 최대 6억 대출).
- **이 가구에 대한 함의**: 실거주 목적이라 2년 의무 부담은 작다. 다만 **전세 낀 매수(갭투자) 불가** + **대출 한도 축소**가 실질 제약이며, 11억대 매수 시 자금계획에 영향을 준다.
- **변별력 주의**: 서울은 전역이 토허제라 **'토허제 비해당'이 서울 내 구(區) 비교 기준이 되지 못한다**. 서울 구 비교는 통근·예산·세대수로만 한다.
- **우리 후보 중 비해당(예외)**: **용인 기흥구**(광교 풍경채 어바니티=흥덕지구), **수원 권선구**(매교역 팰루시드=세류동). 그 외 서울 전 매물·분당·수지·팔달·영통 매물은 모두 해당. → `listings.json`의 매매 `토허제 해당` 컬럼은 이 기준으로 관리한다.

## 매매 가격 컬럼: 호가·실거래가 (국토부 API)

매매 표에는 가격 컬럼이 둘이다. **둘은 반드시 같은 전용면적끼리 비교**한다(면적이 다르면 비교 무효). **AI 에이전트가 매물을 추천·갱신할 때 두 값을 아래 정의대로 채운다.**

- **`호가(억)`**: **현재 웹상에 존재하는, 거래 진행 예정인(매물로 나와 있는) 호가.** 즉 지금 네이버 부동산 등에 올라온 매도 호가다. 협상은 실거래가가 아니라 이 호가를 베이스로 시작하므로 실무상 매우 중요하다. 호가는 공공 API가 없어 `scripts/naver_pull.py`(네이버 모바일 비공식 경로)로 받거나 사용자가 직접 입력한다. (월세/전세 표에는 없는 컬럼.)
- **`최근 실거래가(억)`**: **국토부 실거래가 API로 조회된, `호가`와 동일한 전용면적의 가장 최근 실제 거래가.** 출처는 국토부 오픈API(아래)다.
- **조회되지 않으면 빈칸으로 둔다.** 해당 전용면적의 실거래가 국토부 API에서 조회되지 않거나, 거래가 비정상(직거래성 이상치)·재건축 입주권 등 **확실한 구축 매매가 아니면 값을 쓰지 않고 빈칸**으로 남겨 사용자가 직접 채우게 한다. 추정·보간·타 출처 대체 금지.

### 실거래가 조회 (국토부 오픈API)

- **인증키는 `.molit.json`의 `service_key`에서 읽는다**(또는 환경변수 `MOLIT_SERVICE_KEY`). 시크릿이라 커밋 금지(`.gitignore` 등록됨). 키 발급은 사람만 가능하며 [노션.md](노션.md)의 토큰과 동일한 보안 원칙으로 취급한다.
- 엔드포인트(승인된 **기본** 매매 자료): `https://apis.data.go.kr/1613000/RTMSDataSvcAptTrade/getRTMSDataSvcAptTrade` — 상세(`...Dev`)가 아니라 **기본** API다(키가 기본에만 승인됨). 기본 API에도 전용면적·거래금액이 들어 있어 면적 매칭이 가능하다.
- 주요 파라미터: `serviceKey`, `LAWD_CD`(시군구 법정동코드 5자리), `DEAL_YMD`(거래연월 `YYYYMM`). 응답(XML)의 `아파트`(단지명)·`전용면적`·`층`·`거래금액`(만원)·`년/월/일`로 **단지명 + 추적 전용면적**을 필터해 가장 최근 거래를 고른다. 국토부 단지명은 우리 표기와 다를 수 있으니(예: `관악드림타운(삼성·동아)`→`관악드림(삼성)`/`관악드림(동아)`) 이름+면적으로 매칭한다.
- **호출 경로**: 직접 `curl`은 WAF에 막히지만(`Request Blocked`), **`scripts/molit_pull.py`(파이썬 urllib)는 Claude Code 세션 안에서도 통한다** — 실거래가 자동 채움은 이 스크립트로 한다(아래). 세션에서 수동 조회만 할 땐 `WebFetch` 도구도 가능. 사용자 로컬 터미널에서는 `curl`도 정상.
- 우리 후보 단지의 `LAWD_CD`:

  | 구 | LAWD_CD |
  |---|---|
  | 서울 관악구 | 11620 |
  | 서울 광진구 | 11215 |
  | 서울 동작구 | 11590 |
  | 서울 중랑구 | 11260 |
  | 성남 분당구 | 41135 |
  | 성남 수정구 | 41131 |
  | 용인 기흥구 | 41463 |
  | 용인 수지구 | 41465 |
  | 수원 권선구 | 41113 |

- 입주 전 분양권(예: 매교역 팰루시드)·민간임대(예: 광교 풍경채 어바니티)는 매매 실거래가 존재하지 않으므로 '거래 없음'으로 둔다.

### 실거래가 자동 채움: `scripts/molit_pull.py`

매매 단지의 `최근 실거래가(억)`을 국토부 API로 자동 채우는 puller다(`notion_pull.py`와 형제). 매칭표는 `data/molit_map.json`(단지명→`lawd_cd`·국토부 단지명 패턴(`aptNm`)·`areas`(평형별 전용면적); `skip` 있으면 건너뜀).

```bash
cd scripts
python3 molit_pull.py            # 최근 6개월 조회→면적 매칭→listings.json 기록
python3 molit_pull.py --dry-run  # 적용 없이 제안만 출력
python3 molit_pull.py --months 8 # 조회 개월 수 조정
python3 make_notion.py           # 이어서 Notion 배포
```

- 단지·전용면적별 **가장 최근 체결가**를 기록한다. 같은 평형 거래의 **중앙값 대비 0.7~1.6배를 벗어나는 직거래성 이상치는 자동 제외**(표본 4건 이상일 때).
- **빈칸 유지** 대상: 매칭 실패 단지, `molit_map.json`에 `skip`으로 표시된 단지(재건축 입주권·오피스텔 의심·민간임대·거래 희소 등). 추정·보간하지 않는다.
- 새 매물을 추가하면 `molit_map.json`에 매칭 항목을 넣어야 자동 채움 대상이 된다. 국토부 표기를 모르면 `aptNm` 패턴을 넓게 잡고 결과를 보며 좁힌다.

### 호가 자동 채움: `scripts/naver_pull.py`

매매 단지의 `호가(억)`을 네이버에서 채우는 puller다. 호가는 공공 데이터가 없고 네이버는 약관상 자동수집을 금지하므로 **회색지대 도구**다 — 개인용·소량 전제로만 쓴다. (로그인하지 않으므로 계정 리스크는 없다.)

작동 방식(검증 완료):

1. `new.land.naver.com` 홈을 한 번 방문해 **쿠키(`REALESTATE` 등)를 얻는다.** 쿠키 없이 곧장 API를 치면 데이터센터 IP는 `429`가 난다 — 쿠키 워밍업이 이를 푼다. (그래서 Claude Code 세션 안에서도 동작한다. 네이버가 막는 건 UA가 아니라 쿠키 없는 콜드 요청이었다.)
2. 단지 페이지 HTML에 박혀 있는 **Bearer 토큰(JWT, 유효 3시간)**을 정규식으로 추출한다.
3. 그 토큰으로 `api/articles/complex/<hscpNo>` 를 호출해 매매 매물을 받고, **평형별 최저 호가**(협상 베이스)와 매물 건수를 기록한다.

> m.land 모바일 ajax(`getComplexArticleList`·`cluster/ajax/articleList`)는 비브라우저 요청에 **조용히 `null`을 반환**(소프트 차단)하므로 쓰지 않는다. new.land + 토큰이 유일하게 안정적인 경로다.

- 매칭표는 `data/naver_map.json`: `{단지명: {"hscpNo": "<단지번호>"(여럿이면 배열), "areas": {평형: 전용면적㎡}, "skip": 사유}}`. `areas`는 실거래가(`molit_map.json`)와 **같은 전용면적**으로 맞춘다. 응답의 `area2`(전용)가 정수로 잘려 와도 25↔34평 차이가 커서 `AREA_TOL=2.0`으로 구분된다.
- **hscpNo는 `--resolve`로 자동 채운다**(new.land 검색 → 단지명+행정구로 매칭). 후보가 여럿이면 직접 고른다. hscpNo는 단지 페이지 URL `new.land.naver.com/complexes/<hscpNo>`에서도 확인된다.

```bash
cd scripts
python3 naver_pull.py --resolve             # 빈 hscpNo를 검색해 naver_map.json에 채움
python3 naver_pull.py --dry-run             # 호가 조회→평형 매칭, 적용 없이 제안만
python3 naver_pull.py                        # listings.json '호가(억)' 기록
python3 naver_pull.py --only "관악드림타운(삼성·동아)"  # 한 단지만
python3 naver_pull.py --sleep 2.5           # 요청 간 대기(초, 기본 1.5) — 429 나면 늘림
python3 make_notion.py                      # 이어서 Notion 배포
```

- 토큰이 만료(401)되거나 429가 나면 다시 실행한다(매 실행마다 새 토큰·쿠키를 받는다). 네이버 응답 구조가 바뀌면 `fetch_sale_articles()`의 필드(`area2`·`dealOrWarrantPrc`·`tradeTypeName`)를 맞춘다.

## 작업 방법

### 0. (필수) 작업 전 Notion pull 선행
매물을 **탐색·추천·추가·수정하거나 Notion에 배포하기 전에는 항상 먼저** Notion의 최신 편집을 회수한다. 사용자가 Notion에서 직접 고쳤을 수 있으므로, 건너뛰면 그 편집을 덮어쓰게 된다.
```bash
cd scripts
python3 notion_pull.py     # Notion 표 → data/*.json merge (로컬 필드 id·estimate·좌표는 보존)
```
`.notion.json`의 토큰·`block_ids`가 아직 없으면(최초 배포 전) pull은 불가하므로 생략한다.

### 매물 추가·수정·표 이동
`data/listings.json`과 `data/config.json`을 고친다.
- 매물 추가/수정: `data/listings.json`의 객체 편집. 매물명을 키로 하고, 각 매물에 안정적 `id`(이름이 바뀌어도 추적할 키)를 부여한다. 장점/단점은 배열로 두면 표에서 자동으로 글머리 처리된다. **매물명은 [매물 항목 원칙]대로 실재하는 특정 단지 하나를 정확히 가리켜야 한다**(클러스터·일반 서술 금지).
- 표(거래유형·시·구) 배치·이동: `data/config.json`의 `groups[].cities[].districts[].names` 배열에 단지명을 넣거나 옮긴다. 새 구는 해당 시의 `districts`에 `{"district": "...", "names": [...]}`를, 새 시는 해당 `group`의 `cities`에 `{"city": "...", "districts": [...]}`를 추가한다(구 표 순서 = `districts` 배열 순서, 시 순서 = `CITY_PRIORITY`). 매물 자체에는 시·구 필드를 두지 않는다 — 배치는 config로만 표현한다.
- 열(속성) 추가·변경: 표마다 컬럼이 다르므로 `config.json`의 **해당 거래유형 `attrs`**만 고친다(예: 매매의 `토허제 해당`, 월세/전세의 `월세 가격`·`전세 가격`). 같은 거래유형의 모든 시·구 표는 `attrs`를 공유한다. 데이터 키는 표시 라벨과 **동일**해야 pull 라운드트립이 맞는다(예: 데이터에 `"가격(억)"` 키로 저장).
- **메타 필드**: **시세(가격)**가 추정치인 매물은 `"estimate": true`를 넣는다(향후 SPA가 ⚠️ 배지로 활용). `id`/`estimate`는 표에 렌더링되지 않는다. 단 [매물 항목 원칙]대로 **세대수·연식·전용면적은 추정이 아닌 실제 단지 값**이어야 한다(이 값들이 범위·미상이면 특정 단지가 아니라는 신호).
- **`AI 추천` / `검토 여부` 컬럼**: Claude가 추천해 추가한 매물은 `AI 추천`에 `O`를 넣는다. `검토 여부`는 사용자가 실물·시세를 직접 확인한 뒤 채우는 칸이므로 **비워 둔다(기본값 없음)**. 정의 안 된 속성은 `cell_text()`가 빈 값으로 처리하므로 기존 매물에 빈 필드를 넣지 않아도 된다.

### 구조 무결성 검증
`scripts/listings.py`의 `validate()`가 config 배치와 listings.json의 정합성을 점검한다: ① config엔 있으나 listings.json에 정의 없는 매물, ② 둘 이상의 구 표에 중복 배치된 매물, ③ 어느 구 표에도 배치 안 돼 Notion에 안 그려지는 매물을 경고한다. (과거 '실거주 25평 미만' 평수 검증은 '실거주'+'재건축 투자'가 '매매'로 통합되며 근거가 사라져 제거했다.)
```bash
cd scripts
python3 listings.py     # 단독 실행 → 경고 목록 출력
```
`make_notion.py`는 생성 직전 자동으로 `validate()`를 돌려 경고를 출력한다(경고는 생성을 막지 않음).

> 수정한 데이터를 Notion에 반영하는 방법은 [노션.md](노션.md) 참고.
